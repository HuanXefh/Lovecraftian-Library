Lovec Lib Guide

LovecLib: A Guide Book


LovecLib is a javascript mod that provides some methods for mostly my other mods. However, any other javascript mods can utilize it if the author is familiar enough with JS (and stuck in javascript like me). This document aims to explain how the lib works and some concepts brought along with it.

Note that the lib is still subject to change, and the document may not be updated that frequently. Some information can be inaccurate or even outdated.

Introduction

Library Structure

Javascript files in Lovec are categorized based on their purpose. At the first glance it may be way more complex than most js mods, but you don't need all of them at most time. Just know what files to check and import when you need something.

Category Use Note
aux Various auxiliay files, including recipes for multi-crafters.
blk Template for blocks. A type of template.
cls Definition for various classes. Advanced users only.
ct Where {extend} is used and contents are created. Should be loaded in your main.js only.
db Stores static parameters. Has cross-mod interaction.
env Template for environmental blocks. A type of template.
frag Collection of game related methods, including component methods.
glb Stores global parameters and objects.
lib External library files.
math Deals with mathematical problems.
mdl Collection of more generic methods.
mod Deals with cross-mod interactions.
rs Template for items and fluids. A type of template.
run Codes usually run only once. Not expected to be imported except RUN_version.
sta Template for status effects. A type of template.
tp Stores static objects.
unit Template for unit types. A type of template.
wea Template for weathers. A type of template.

Extending A Template

To extend a template, you should find a template file first. Template files serve as collections of pre-defined methods. As you can see in Library Structure, files in blk folder are templates for blocks, every {BLK_xxx} is a template file that you can extend to create your own contents.

Template files are meant to be used in a content file (CT file), which is in the ct folder. A CT file may consists of three parts in two sections: import (definition), auxiliay (definition) and application. Application is simply where you write {extend(..., {...})}, I won't talk about it since that's basic knowledge for a javascript modder.

In the import part, besides importing modules you need, it's important to set the right TEMPLATE. If you are creating conveyor blocks, write {const TEMPLATE = require("lovec/blk/BLK_conveyor")} or similiar line to require your own conveyor template. Generic methods for a type of contents should always be defined in a template file, NOT the content file, especially in the object in extend which is completely not reusable. If you're doing it right, you are expected to import few files in a CT file, on most occasions TEMPLATE is the only one required.

Most CT files in ProjReind have auxiliay part, where {_std} and {_std_b} are defined. The two methods are local and responsible for outputting objects used in extend. Some objects (e.g. in {CT_BLK_recipeFactory}) are really long and if you put them in extend instead of using a function, you'll get f**ked up by the huge gap between each content, and it's torture to edit every single object when something changes.

I can't export {_std} (sounds kinda wierd). CT files are only meant to be required in the scope of your own mod, if you use require on them in cross-mod situations something may break. For your own CT files, you have to copy {_std} or write your own ones.

You may have found that {this.super$xxx} is ommited in some methods randomly. In the template file, methods tagged with @NOSUPER are expected to be used with the super method not called.



Functionality

Class

The class keyword is reserved for javascript mods. We have to use the primitive form of class in early js, the Function. In Lovec, functions are divided into methods and function classes. Let {fun} be a function, then {fun.isClass()} will return false if it's a method, or true if function class.

To create a function class:


        const YourClass = function() {
          this.init.apply(this, arguments);
        }.initClass();

        YourClass.prototype.init = function(...) {
          ...
        };

The {initClass} method called on {YourClass} is required to turn it into function class. The {init} on the prototype, however, is used to set up the behavior of the constructor. Once you have created the class, you can simply create instances by {new YourClass(...)}. To define methods (or properties) for the class and its instances:


        YourClass.testClassMethod = function(...) {
          // {this} here refers to {YourClass}
          ...
        };

        YourClass.prototype.testInstanceMethod = function(...) {
          // {this} here refers to the instance
          ...
        };

It's quite simple if you know how prototype works. If you'd like to export the class:


        // In class file
        module.exports = YourClass;

        // In other file
        const YourClass = require(...);
        let ins = new YourClass(...);

At most time classes inherit methods from a parent called super class. To do that in Lovec:


        const YourClass = function() {
          this.init.apply(this, arguments);
        }.extendClass(*your super class*).initClass();

        YourClass.prototype.init = function(...) {
          // Can be ommited if no need
          this.super("init", ...);
          ...
        };

The {extendClass} method should always be called BEFORE {initClass}. If there's a super class, the {super} method can be used in class methods and instance methods to call the corresponding functions defined in the super class, just like:


        YourClass.testClassMethod = function(arg1, arg2, arg3) {
          // Use arguments that the super method actually needs!
          this.super("testClassMethod", arg1, arg2, arg3);
          ...
        };

If you leave the method untouched, it will be exactly the same as the counterpart in the super class.


Coding Tags

To avoid abuse of note, tags are used to mark some objects. Tags are used in note and start with "@" (NOT annotation which is defined in another way), here's a list of tags and their meaning:

Tag Meaning Description
@ARGS In any file.
Note of function arguments when not explicitly shown in the definition.		 Possible arguments.
@CONTENTGEN In DB file.
The array receives elements later, which are added by codes.
@DYNAMIC In DB file.
Getter functions can be used here for dynamic value.		 Getter format.
@FIELD Mostly in component method.
These fields should be defined to use the method.		 Field names.
@IMPLEMENTED In class file.
This method is imported from an interface.
@INHERITED In class file.
This method is inherited from the super class, {this.super} can be used.
@LATER In class/instance method.
Definition of this method should be done in child classes.
@METHOD Mostly in component method.
These methods should be defined to use the method.		 Method names.
@MULTIEXTEND In template file.
Two ore more methods are overwritten in extend, {arguments} should be passed down.
@NAMEGEN In template file.
The content name will be generated if not set in bundle.
@NOSUPER In template file.
{this.super$xxx} should not be used in the content files.
@NULL In any file.
This parameter can be null.
@PARAM In template file.
The key is expected to be set by the modder.		 Use of the parameter.
@SINGLESIZE In template file.
The block is expected to be one block large, set size to 1.
@SPEC In content file.
The lines below are special and for this content only.
@TABLE In any file.
Note for table parts.		 Part name.
@VERSION In any file.
The method or property is changed after this version.		 Lovec version.

Annotation

It's totally different from the one used in Java. In Lovec, annotation is used to modify a function. For example, {__SERVER__} will make a method only runs on server side. Annotations in Lovec are defined in {BOX_anno} which is in glb, the annotation class is defined in {CLS_annotation}.

Annotation may appear to have arguments. Take {__INIT__} as an example, the sole argument is a function which is called after the definition of the parent function (as the 3rd argument in {setAnno} since it's loading argument):


        const foo = function() {
          ...
        }
        .setAnno(ANNO.__INIT__, null, function() {
          ...
        });

{BOX_anno} is usually imported as {ANNO} since it's in glb. To apply annotation to a function, simply call {setAnno} on it. However the function should not be directly declared with the function keyword, as {setAnno} works by returning the modified version of your function.

At most time, {this} in the annotation (including the function used in {__INIT__}) refers to the parent function. However, inside an arrow function {this} will always be the global object, which is why I discourage using arrow functions in annotation-related codes.


Auto-merge

Auto-merge is used for DB files. Lovec by default only reads DB files defined locally, to access the data from other mods, DB files in Lovec will simply search other DB files in loaded mods and try extending their arrays with copied data.


        // Defined in "lovec/scripts/db/DB_test"
        const db = {
          "arr": [0, 1],
          "obj": {
            "arr1": [0, 1],
            "obj1": {
              "arr2": [0, 1],
            },
          },
          "str": "ohno",
        };


        // Defined in "your-mod/scripts/db/DB_test"
        const db = {
          "arr": [2, 3],
          "obj": {
            "arr1": [0, 1],
            "obj1": {
              "arr2": [4, 5],
              "arr3": [6, 7],
            },
          },
          "num": 7,
        };


        // The object generated after auto-merge
        const db = {
          "arr": [0, 1, 2, 3],
          "obj": {
            "arr1": [0, 1, 0, 1],
            "obj1": {
              "arr2": [0, 1, 4, 5],
            },
          },
          "str": "ohno",
        };

{db} in DB files should contain objects and arrays only, which are the targets of auto-merge. Strings, numbers and functions are not merged, always put them in an array. Since it's only called in Lovec DB files, it only merges fields that DBs in Lovec have (in the code above "arr3" is ommited).

Auto-merge only supports a maximum of 3-layer depth, it's impractical to dig deeper on most occassions. In the {db} below "arr2" in "obj2" is too deep to merge.


        const db = {
          "obj": {
            "arr": [],
            "obj1": {
              "arr1": [],
              "obj2": {
                "arr2": [],
              },
            },
          },
        };

Version Check

You can't simply require a mod as the dependency when it's updated often. As the mod updates, methods can be changed and sometimes that will lead to random crash. It's important to tell the user they are using outdated mods.

Lovec has provided a way to check if the currently loaded mods meet the minimum version requirements, which is defined in RUN_version. For example, if you'd like to require Lovec v101 and Lovec Lab v101, you will need to put this at the top of your main.js.


        if(require("lovec/run/RUN_version").checkVersion(*your mod name*, [
          "lovec", 101,
          "loveclab", 101,
        ])) return;

It will prevent loading of your mod scripts and throws an error dialog to inform the user of outdated or missing mods.

Version here takes format of "xxx.xxx" for convenience of comparison, digits after the decimal point are used for beta versions (e.g. 100.25063001, use date to make it larger for newer versions). If you need running version check on your own dependencies, the version stat in mod.json or mod.hjson cannot be like "xxx.xxx.xxx" or contain letters.

However, if you don't create a library, it does not matter how you write the mod version, since it's not a dependency.



Important Files

RUN_jsExt

MDL_event

This module handles events, methods here are just encased {Events.run(...)} or {Events.on(...)}.

It's important to know that, if required cross-mod, a script can be run multiple times. For example:


        Events.run(ClientLoadEvent, () => print("ohno"));
        MDL_event._c_onLoad(() => print("ohsh"));
        MDL_event._c_onLoad(() => print("ohyes"), 15227481);

If required by 3 mods, after client load you'll find 3 "ohno"s, 3 "ohsh"s and 1 "ohyes" in the console. The numeric id used here prevents repetition, which is applied to all methods defined in {MDL_event}. Although set manually in Lovec, you can utilize {Math.intUnique} to generate the id which is defined in {RUN_jsExt_util}.

Recipe Factories

Lovec has javascript-based multi-crafters, see BLK_recipeFactory and its child templates.

Recipe factories generally require recipe files which are put in "scripts/aux/rc". A recipe file should be named after its owner building (more precisely, the internal name), without mod name prefix. For the format of a recipe file, you can check the files in Project Reindustrialization.

Most parameters are included in the "recipes" array, which is a 2-array comprised of recipe header and recipe object. Recipe header is a string that is unique to a recipe, it replaces the old numeric id used in Reind. Recipe object is the main place where a recipe is defined, here's a table for its fields (you don't need all of them):

Field Description Note
icon The icon shown for this recipe in config table.
Expected to be a content name.
category Category the recipe is in.
isGenerated Whether the recipe is generated by codes. Do not manually set this.
validGetter A boolean function that determines whether the recipe is available now.
Format: b => {...}
lockedBy Recipe is locked until these contents are unlocked/researched. Always an array!
timeScl Multiplier on the final crafting time.
ignoreItemFullness Wether the building continues consuming when full of item ouputs.
tempReq For BLK_furnaceFactory only.
The furnace temperature required to start crafting.
tempAllowed For BLK_furnaceFactory only.
The maximum temperature allowed for the recipe.
durabDecMtp For BLK_durabilityFactory only.
Multiplier on the durability decrease rate.
tooltip The extra info used in tooltip.
Note that it's a bundle piece, do not write text here!		 info.common-info-tt-*value*.name
I/O fields
ci Continuous input. Fluids only.
bi Batch input.
aux Auxiliary input. Better for auxiliay fluids only.
reqOpt Whether at least one optional input is needed for crafting.
opt Optional input. Items only.
